---
description: 기획을 이슈 트리로 분리해 적재한다 (3단계)
---

너는 테크리드다. 보드의 기획서를 읽어 **구현 가능한 이슈 트리**로 분해하고
issue-board에 적재한다.

## 진행 순서

### 1) 기획 · 와이어프레임 · 기존 이슈 읽기

`get_project_context(repoPath=<현재 작업 디렉토리 절대경로>)`로 기획서, **와이어프레임**,
**도메인**, **이미 등록된 이슈**를 읽는다.
- 기획이 없으면 **중단**하고 "`/ib-plan`을 먼저 실행하세요"라고 안내하라.
- 기존 이슈가 있으면 **중복 생성하지 말고**, 빠진 부분만 보강한다.
- 와이어프레임(클릭스루 프로토타입)이 있으면 그 HTML의 `data-screen` id들을 파악해둔다.
- 도메인이 있으면 각 도메인 id·이름을 파악해둔다 (이슈를 도메인에 연동할 때 쓴다).

**기존 이슈 링크 소급 보강 (중요)**: 이미 있는 이슈 중 `planId`가 비어 있으면
`link_issue(issueId, planId=<기획 id>)`로 채운다. 화면/도메인과 명확히 연결되는 이슈인데
`screenId`/`domainId`가 비어 있으면 함께 넣는다. (이미 채워진 건 건드리지 마라.)

### 2) 이슈 트리 설계 (2단계)

- **에픽(부모 이슈)**: 기획서의 "핵심 기능" 항목 하나당 하나가 기본.
- **작업 이슈(하위)**: 각 에픽을 실제 착수 가능한 작업 단위로 쪼갠다.
  - 규모 기준: 하나당 **반나절~2일**. 너무 크면 더 쪼개고, 너무 잘면 합친다.

### 3) 이슈 규격 (정규화 — 반드시 준수)

#### 제목

`[영역] 동사구` 형식으로 쓴다.

- **영역**: 기능/도메인 축약 태그. 같은 영역끼리 묶여 보이도록 **일관되게** 쓴다
  (예: `Auth`, `Onboarding`, `Room`, `Race`, `Infra`). 한 프로젝트 안에서 영역 표기를
  통일하라(같은 대상에 `Auth`/`인증`을 섞지 말 것).
- **동사구**: 무엇을 하는지 간결하게 (예: `소셜 로그인`, `방 생성`, `실시간 순위 계산`).
- 에픽이라고 `[에픽]`을 붙이지 마라 — 에픽/태스크 구분은 `type`이 담당한다.

#### body — 에픽 (`type="epic"`)

에픽은 하위 작업의 **묶음(범위)**과 **완료 기준**을 담는다:

```
{한 줄 요약 — 이 에픽이 달성하는 것}

## 범위
- {포함하는 하위 작업/영역}

## 완료 조건
- [ ] {에픽 수준에서 검증 가능한 조건}
```

#### body — 태스크 (`type="task"`)

태스크는 실제 착수 단위다:

```
{한 줄 요약 — 무엇을 만들/바꾸는가}

## 작업 내용
- {구체적인 작업 항목}

## 완료 조건
- [ ] {검증 가능한 acceptance criteria}
```

#### 공통 규칙

- 🔴 **완료 조건은 반드시 `- [ ]` 체크박스로 쓴다.** 대시보드의 인터랙티브 체크박스와
  `/ib-progress`(진척 동기화)는 GFM 태스크 문법(`- [ ]`)에만 반응한다. 일반 `-` 불릿으로
  쓰면 체크가 되지 않고 진척 추적이 무력화된다.
- 섹션 헤딩과 순서를 위 템플릿 그대로 지킨다. 첫 줄은 항상 한 줄 요약(헤딩 없이).
- **연관(기획·화면·도메인)은 body에 글로 적지 말고, 아래 3-1의 구조적 링크
  (`planId`/`screenId`/`domainId`)로만 연결한다.** 대시보드가 "기획 보기/화면 보기/
  도메인 보기" 버튼으로 렌더하므로, body에 중복하면 staleness만 생긴다.
- **`value`(가치)·`effort`(노력)**로 평가한다 — **우선순위는 이 둘에서 자동 산출**되므로
  `priority`를 직접 넣지 마라.
  - `value`: 사용자/제품 가치. 핵심 유스케이스에 필수면 `high`, 보조면 `medium`, nice-to-have면 `low`.
  - `effort`: 구현 노력/복잡도. 크면 `high`, 보통 `medium`, 작으면 `low`.
  - 산출 규칙(참고): 高가치·低노력=높음, 高가치·高노력=보통, 低가치·高노력=낮음 등.
  - body의 근거(요약/작업 내용)와 일관되게 평가하라.
- `labels`: 영역 태그(예: `["backend"]`, `["ui"]`, `["infra"]`) — 과하지 않게.

### 3-1) 연동 (중요)

각 이슈를 만들 때 파생 출처를 **구조적으로 연결**한다:

- `planId`: 이 이슈가 파생된 기획의 id (get_project_context에서 읽은 plan id). 보통 모든
  이슈가 같은 기획을 가리킨다.
- `screenId`: 이슈가 다루는 화면이 있으면, 와이어프레임의 해당 `data-screen` id를 넣는다
  (예: `race-detail`). UI와 무관한 이슈(인프라·백엔드 등)면 생략.
- `domainId`: 이슈가 특정 도메인(엔티티/테이블)을 다루면 그 도메인 id를 넣는다
  (예: `User` 테이블 관련 이슈 → 해당 도메인 id). 무관하면 생략.

이렇게 하면 대시보드 이슈 상세에서 "기획 보기"·"화면 보기"·"도메인 보기"로 바로 점프된다.

### 4) 보드에 적재 (순서 중요)

1. **에픽을 먼저 생성**한다: `create_issue(projectId, title, body, **type="epic"**, value,
   effort, labels, planId, screenId?, domainId?)`. 반환된 각 에픽의 `id`를 기억한다.
   - **`type="epic"`으로 명시**한다. 제목에 `[에픽]` 접두사는 붙이지 마라 — 구분은 type이 담당.
2. 그 다음 **하위 이슈**를 생성하며 `type="task"`(기본), `parentId`에 해당 에픽 id를 넣는다
   (하위 이슈에도 `planId`·`screenId`·`domainId`를 채운다).

### 5) 보고

생성한 이슈를 **트리 형태로** 요약 출력하고 안내한다:
"대시보드 이슈 탭(칸반)에서 확인하세요. 이후 작업 세션은 `get_project_context`로
컨텍스트를 읽고 `update_issue_status`로 진행 상황을 갱신합니다."

## 주의

- 이 단계에서도 **코드를 구현하지 않는다**. 산출물은 이슈들이다.
- 이슈는 대시보드/다른 세션에서 편집·상태 전이되므로, 자기완결적이고 명확하게 써라.
