# 공통 규약 (모든 ib-* 생성 스킬)

이슈보드에 산출물을 적재하는 모든 스킬이 공유하는 전제·규칙. 각 스킬은 이 파일을 개별로
반복하지 않고 여기서 참조한다.

## MCP 전제

- 이 스킬군은 issue-board MCP 서버(`http://localhost:4000/mcp`)에 의존한다.
- **MCP 툴이 안 보이면** 대상 프로젝트 루트에 `.mcp.json`이 있는지, 서버가 떠 있는지 먼저
  확인하고 안내하라. (툴 없이는 진행 불가.)

```json
{ "mcpServers": { "issue-board": { "type": "http", "url": "http://localhost:4000/mcp" } } }
```

## 프로젝트 매칭 · 생성

- 시작 전 **항상** `get_project_context(repoPath=<현재 작업 디렉토리 절대경로>)`로 이미 등록된
  프로젝트·산출물이 있는지 확인한다.
- 프로젝트가 없으면 `create_project(name, description, repoPath=<cwd 절대경로>)`.
  - `repoPath`는 **반드시 현재 cwd의 절대경로**를 쓴다. 다른 세션이 cwd로 프로젝트를 매칭한다.
- 이미 기획/이슈가 있으면 **중복 생성하지 말고**, 빠진 부분만 보강하거나 사용자에게
  새로 만들지·이어서 보강할지 물어라.

## draft / upsert / status

- 도메인 · 디자인은 **이름/프로젝트 기준 upsert** — 같은 이름으로 다시 호출하면 갱신(중복 아님).
- **첫 설계는 status를 생략**해 자동으로 **`draft`(초안)**으로 둔다. 대시보드에 "초안" 배지가 붙는다.
- 확정 요청을 명시적으로 받았을 때만 `status="approved"`.

## 완료 조건 체크박스 (🔴 필수)

이슈 본문의 "완료 조건"은 **반드시 GFM 태스크 체크박스 `- [ ]`** 로 쓴다. 대시보드의
인터랙티브 체크박스와 `ib-progress`(진척 동기화)는 `- [ ]` 문법에만 반응한다. 일반 `-`
불릿으로 쓰면 체크가 되지 않고 진척 추적이 무력화된다.

## 산출물 원칙

- **코드를 구현하지 마라.** 이 스킬군의 산출물은 기획 · 와이어프레임 · 도메인 · 이슈 ·
  디자인 토큰이다 (마이그레이션 · 엔티티 클래스 · 컴포넌트 코드 아님).
- **삭제(`delete_*`)는 사용자가 명시적으로 요청할 때만.** 임의로 호출하지 마라.
- 재생성은 대개 **새 버전 / upsert**이지 삭제가 아니다.
