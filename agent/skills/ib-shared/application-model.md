# 애플리케이션(전달 표면) 모델 · 이슈 키

한 프로젝트는 여러 **애플리케이션**(전달 표면)으로 나뉠 수 있다 (예: `추노앱` 사용자 앱과
`백오피스` 관리자 콘솔). **기획 · 와이어프레임 · 이슈는 앱 단위로 묶이고, 도메인(데이터
모델)은 앱들이 공유한다.**

## 규칙

- **모든 이슈는 정확히 하나의 앱에 속한다 (앱 필수).** 프로젝트가 개념상 단일 앱이어도
  **기본 앱 하나를 확보**해 그 앱에 붙인다 — 그래야 모든 이슈에 예외 없이 키가 부여된다.
- `get_project_context`의 `applications`를 보고 이 작업이 **어느 앱**에 속하는지 정한다.
  - 기존 앱에 속하면 그 앱의 `id`를 쓴다.
  - **새 전달 표면**이면(관리자용 백오피스, 별도 웹 등) 아래 `create_application`으로 만든다.
  - 앱이 하나도 없으면 **기본 앱 하나를 만들어** 확보한다 (사용자에겐 거의 안 보이지만 키의 근거가 된다).
- 확보한 **`applicationId`를 create_plan · create_wireframe · create_issue에 모두 넘긴다.**
  (도메인은 공유이므로 `applicationId`를 주지 않는다 — 백오피스도 같은 User · Room을 다룬다.)

## create_application

```
create_application(projectId, key, name, description, issuePrefix)
```

- `key`: 안정적 식별 키(kebab-case, 예: `chuno-app`, `backoffice`). 프로젝트 내 유일.
- `name`: 표시명 (예: "추노앱", "백오피스").
- `issuePrefix`: **이슈 키 접두사 — 대문자 2~4자, 프로젝트 내 유일** (예: `CH`, `BO`).

### 🔴 새 앱을 만들 때는 접두사를 반드시 사용자에게 묻는다

**`create_application`을 호출하기 전에 `AskUserQuestion`으로 이 앱의 이슈 키 접두사를
물어라.** 접두사는 이슈 키(`CH-12`)에 영구히 남고 부여 후 바꾸기 번거로우므로, 자동으로
정하지 말고 사용자가 고르게 한다.

- 앱 이름을 로마자화해 **후보 1~2개를 제안**한다(예: 추노앱 → `CH` / `CHUNO`, 백오피스 → `BO`).
- 사용자가 직접 입력할 수 있게 열어둔다(대문자 2~4자).
- 이미 쓰는 접두사와 겹치면(프로젝트 내 유일 위반) 다시 묻는다.
- 답을 받은 값을 `issuePrefix`로 넘긴다. (서버도 유일성을 검증하고, 충돌 시 오류를 돌려준다.)

## 이슈 키 (`CH-12`)

- 형식: **`<issuePrefix>-<번호>`** (예: `CH-12`, `BO-3`).
- **번호는 앱별 순번**(1부터 단조 증가). 에픽 · 태스크가 **한 카운터를 공유**하며 생성순으로 매겨진다.
- 채번은 서버가 `create_issue` 시 원자적으로 처리한다 — **키/번호를 직접 지정하지 마라.**
  `create_issue` 응답의 `key`를 받아 쓴다.
- **키는 부여 후 불변**이다. 이슈를 다른 앱으로 옮겨도 원래 키(`CH-12`)를 유지한다.

### 키로 이슈 지칭 (토큰 절약)

- MCP read 응답(`get_project_context` · `list_issues` · `get_issue`)은 각 이슈에 `key`를 포함한다.
- 트리 요약 · 보고 · 대화에서 이슈를 **cuid 대신 키로** 부른다 (`CH-12 소셜 로그인`).
- 변이 툴(`get_issue` · `update_issue` · `update_issue_status` · `link_issue`)과 `create_issue`의
  `parentKey`는 **cuid 또는 사람 키(`^[A-Z]+-\d+$`) 어느 쪽으로도** 이슈를 지칭할 수 있다.
  키는 프로젝트 컨텍스트 안에서 해석된다.
